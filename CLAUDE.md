Guidance for Claude Code working in this repository.

## What this is
**ClaimSense AI** — an AI-powered insurance claims intelligence platform. A 10-agent LangGraph pipeline (Intake Supervisor → Policy Coverage → Document Intelligence → [Medical, Billing, Fraud Detection in parallel] → Historical Similarity → Settlement Recommendation → **Human Approval** → Report Generation) that ingests claim documents (policy PDFs, medical records, ID proofs, evidence photos, etc.) and produces a decision-ready package: coverage validation, medical/billing validation, fraud risk score, similar-claim context, a settlement recommendation, a claims-officer decision, and an adjuster-ready PDF report reflecting that final decision. Full spec: `insurance agent.txt`. Visual architecture reference: `claimsence.png`.

## Current status
Backend is functionally complete for the core 9-agent pipeline (Phases 3-9 of `projectfolder.txt`), plus 3 of 4 Phase 10 production-hardening items, plus a Human Approval stage added in a later session. The graph now runs `START -> document -> supervisor -> policy -> medical -> {billing, fraud, history in parallel} -> settlement -> human_approval -> report -> END` ✅ (fan-out/fan-in via LangGraph's native multi-edge pattern, live-verified — see `projectfolder.txt` section 2's Phase 10 entry). Uploaded documents get OCR'd via Sarvam Vision, then four agents make real Sarvam-30B reasoning calls (Policy, Medical, Billing, Fraud) and three run deterministic logic by design (Settlement is a fixed rule cascade, Report renders a real PDF via `fpdf2`) — a deliberate hybrid chosen for auditability in a regulated domain, not a shortcut. Historical Similarity is now real semantic vector search against Qdrant Cloud ✅ (FastEmbed local embeddings, no extra API key — replaced the earlier local `difflib` approach; live-verified cross-run persistence). Settlement now deterministically deducts the policy's deductible and copayment percentage (parsed from free text via regex, with safe skip-don't-guess fallback for unparseable text) from the recommended amount ✅ — unit-verified, not yet exercised in a live full-pipeline run reaching approve/need_review.

**Human Approval ✅** — `POST /claims/{id}/process` now pauses (via LangGraph's `interrupt()`) right after Settlement Recommendation and returns `status: "awaiting_approval"` with every finished agent result but no report yet — the pipeline no longer auto-finalizes a claim outcome. A claims officer (currently the same signed-in user — self-approval demo pattern, no role system exists, see the scoping note below) reviews the AI's recommendation and calls the new `POST /claims/{id}/decision` (`approve`/`reject`/`modify`, with an optional amount override and notes) to resume the graph through Report Generation, which now includes both the AI's recommendation and the officer's final decision in the PDF. Test coverage for this pause/resume/report path is network-free (`backend/tests/test_claims_workflow.py`, zero uploaded documents — no `SARVAM_API_KEY` needed) and passes. See `backend/CLAUDE.md`'s Human Approval entry and `frontend/CLAUDE.md`'s claim submission flow entry for implementation detail.

A claim's final response (after the officer's decision) includes a coverage decision, medical validation, a computed and deducted payable amount, a 0-100 fraud score, real similar-claim context from Qdrant, an approve/reject/need-review settlement recommendation, the officer's final decision, and a downloadable adjuster-ready PDF (`GET /claims/{id}/report`). All of this is live-verified against real Sarvam API calls, a real Qdrant Cloud instance, and a real synthetic claim packet (policy PDF, discharge summary, prescription, lab report, hospital bill) — not mocked; the Human Approval stage itself is verified network-free (see above), not yet re-run against a live Sarvam claim end-to-end. Still no Convex/DB persistence of claim/document *metadata* — re-processing a claim from scratch still re-spends real Sarvam credits (`?force=true`); claim/document metadata resets on server restart (graph *execution* state is Convex-backed via checkpointing, see below). No auth on any endpoint. Convex persistence of claim/document metadata and real auth-backed role separation (beyond today's self-approval demo pattern) are **deliberately queued for later**, not forgotten or blocked. Treat any claim about "what the app does" with suspicion until verified by reading actual code — the docs describe the target, and even this status note can drift from what's actually implemented.

The backend is deployed on Railway at `https://claimsense-ai-production.up.railway.app` (`APP_ENV=production`); all endpoints live under `/api/v1` (e.g. `/api/v1/health`), not at bare paths. UploadThing (file storage) and Convex (auth + persistence + checkpointing) are both deliberately deferred until *after* the Next.js frontend exists — see `backend/CLAUDE.md` for the current wiring status of each.

## Scoping note
This project targets **production-grade engineering practices on demo-scale/synthetic data**, not a certified, regulator-approved production insurance system. Real deployment would additionally require regulatory compliance (IRDAI/HIPAA-equivalent), audited decisioning, legal review of auto-approval logic, and real claims data — none of which are in scope here. Keep this framing in mind when writing docs, code comments, or demo materials: be accurate about what's a working demo vs. what production would additionally require.
