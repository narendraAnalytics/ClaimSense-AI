# frontend/ — CLAUDE.md

Guidance for Claude Code when working in this directory. See the repo-root `CLAUDE.md` for overall project context, `../insurance agent.txt` for the full product/architecture spec, and `../backend/CLAUDE.md` + `../projectfolder.txt` for backend implementation detail.

## What this is
**ClaimSense AI** is an AI-powered insurance claims intelligence platform: a 9-agent LangGraph backend ingests claim documents (policy PDFs, medical records, hospital bills, etc.) and produces a decision-ready package — coverage validation, medical/billing validation, fraud risk score, similar-claim context, a settlement recommendation, and an adjuster-ready PDF report. This `frontend/` directory is the Next.js app that end users (claimants/adjusters) interact with; it talks to the already-built backend over HTTP.

## Deployments
Both halves are now live and can be wired together:
- **Frontend** (this app) — Vercel: `https://claimsense-ai-rust.vercel.app/`
- **Backend** — Railway: `https://claimsense-ai-production.up.railway.app`, all endpoints under `/api/v1` (e.g. `GET /api/v1/health`)

## Backend API surface to build against
The backend's 9-agent pipeline is functionally complete and live-verified (real Sarvam OCR/reasoning calls, real Qdrant Cloud search, real PDF generation). Relevant endpoints:
- `POST /api/v1/claims` — create a claim
- `POST /api/v1/claims/{claim_id}/upload` — upload a document (policy PDF, discharge summary, prescription, lab report, hospital bill, etc.)
- `POST /api/v1/claims/{claim_id}/process` — runs the full 9-agent pipeline synchronously; returns coverage decision, medical validation, computed payable amount, 0-100 fraud score, similar-claim context, an approve/reject/need-review settlement recommendation, and a `report_url`
- `GET /api/v1/claims/{claim_id}/report` — downloads the generated adjuster-ready PDF
- `GET /api/v1/health`

Important caveats when building UI around these:
- `/process` is slow (OCR + 4 real LLM calls) and has **no checkpointing yet** — calling it again on the same claim re-runs the entire pipeline from scratch and re-spends real Sarvam API credits. Treat it as a single long-running action in the UI (e.g. disable re-submit, show progress), not something to poll/retry casually.
- **No auth on any endpoint currently** — anyone with a `claim_id` can call these, including the report download. Don't assume the backend enforces access control; that's frontend-phase work (see below).

These aren't oversights — the backend `CLAUDE.md`/`projectfolder.txt` explicitly queue them for once this frontend exists:
- **Convex Auth** — ✅ done. `@convex-dev/auth` is wired up against the existing prod Convex deployment (`ceaseless-sparrow-692`, the same one `backend/.env` points at — there's no separate dev deployment for this project). Password (email/password) and Google OAuth providers are both live (`convex/auth.ts`). `convex/schema.ts` has `authTables`, `convex/http.ts` registers Convex Auth's HTTP routes, `convex/auth.config.ts` configures the JWT domain. `JWT_PRIVATE_KEY`/`JWKS`/`SITE_URL`/`AUTH_GOOGLE_ID`/`AUTH_GOOGLE_SECRET` are set as Convex deployment env vars (not in any `.env` file — set via `npx convex env set`). `SITE_URL` is currently `https://claimsense-ai-rust.vercel.app` (the deployed site) — Google sign-in **will not work from `localhost` right now**; flip `SITE_URL` back to `http://localhost:3000` temporarily to test Google OAuth locally (Password sign-in is unaffected either way, since it doesn't use `SITE_URL`). `src/app/layout.tsx` wraps the app in `ConvexAuthNextjsServerProvider` + `ConvexClientProvider` (`src/components/providers/convex-client-provider.tsx`); `middleware.ts` runs `convexAuthNextjsMiddleware()` with a broad matcher but no forced redirects yet, since there is no protected route to gate. `src/app/sign-in/page.tsx` + `src/components/auth/sign-in-form.tsx` render the sign-in/sign-up UI. `site-header.tsx` now takes an optional `minimal` prop — when true it renders only the logo/tagline (linked to `/`), hiding nav links and the welcome/avatar block; `/dashboard`, `/claims/new`, and `/claims/[claimId]` all pass `minimal`, so the full nav links + "Welcome, {name}" + avatar/sign-out block is landing-page-only (`/`) now. `hero-section.tsx`'s CTA is auth-aware (`useConvexAuth`): "Start Free Trial" → `/sign-in` when signed out, "Dashboard" → `/dashboard` when signed in — this replaced the header's old "Dashboard" link entirely.

  **New Password sign-ups are routed to `/username`** (`src/app/username/page.tsx` + `src/components/auth/username-form.tsx`) to set a display name (and optional phone, written to the `users` table's existing `phone` column) before reaching `/` — enforced by a `UsernameGate` client component inside `convex-client-provider.tsx` that redirects whenever `useQuery(api.users.current)` has no `name` yet. Google sign-ins already have a `name` from their OAuth profile and skip this step. `convex/users.ts` has the `current` query and `setProfile` mutation backing this.

- **Claim submission flow** — ✅ done. `convex/schema.ts` adds `claims` (`by_user` index) and `documents` (`by_claim` index) tables. `convex/claims.ts` (`create`/`listMine`/`get`/`setBackendClaimId`/`updateStatus`/`saveResults`) and `convex/documents.ts` (`generateUploadUrl`/`insert`/`markBackendUploaded`/`listByClaim`/`remove`/`getUrl`) are ownership-checked the same way `convex/users.ts` is — every function re-derives `getAuthUserId(ctx)` and confirms the claim's `userId` matches, no exceptions. `src/lib/backend-api.ts` is a thin fetch wrapper around the Railway backend (`createClaim`/`uploadDocument`/`processClaim`/`getReportUrl`), reading `NEXT_PUBLIC_API_URL` (new env var, set locally and on Vercel) and appending `/api/v1` itself. Routes: `/dashboard` (`src/app/dashboard/page.tsx`, claim list + "New Claim"), `/claims/new` (`create-claim-form.tsx` — creates the Convex row *and* calls the backend's `POST /claims`, storing the returned `claim_id` as `backendClaimId`), `/claims/[claimId]` (`claim-workspace.tsx` composing `document-uploader.tsx`, `document-list.tsx`, `process-claim-button.tsx`, `claim-results.tsx`, all under `src/components/claims/`). **Uploads go to two places from the browser**, not one: each file is first pushed to Convex File Storage (`ctx.storage.generateUploadUrl()` → `storageId` persisted on the `documents` row, for audit/persistence) and then separately sent as multipart form data straight to the backend's `POST /claims/{id}/upload` (the backend's contract takes raw bytes, not storage IDs — passing IDs instead is a Phase 2/background-job idea from `pdffile.txt`, not today's API). `middleware.ts` now does real route protection — `convexAuthNextjsMiddleware`'s callback context (`convexAuth.isAuthenticated()`) redirects unauthenticated requests to `/sign-in` for a `createRouteMatcher(["/dashboard", "/dashboard/(.*)", "/claims", "/claims/(.*)"])` match, layered on top of (not replacing) the existing client-side `UsernameGate` pattern. `create-claim-form.tsx`'s Policy Number field is now auto-generated client-side (`"POL-" + Date.now().toString(36).toUpperCase()`, mirroring `convex/claims.ts`'s `claimNumber` pattern) and shown read-only — the user never types it, since nothing downstream validates it against a real policy registry (advisory-only LLM context, see `backend/CLAUDE.md`).

  `claim-results.tsx` has a `DECISION_LABELS` map (`approve`→"Approved", `reject`→"High Risk – Manual Review Required", `need_review`→"Manual Review Required") that softens the raw `settlementDecision` string for display only — the stored/API value itself is still the raw `approve`/`reject`/`need_review` string; the tone-color logic still compares against those raw values. Mirrors `_DECISION_LABELS` in the backend's `report_generator.py`.

  `DOCUMENT_TYPES` in `src/lib/backend-api.ts` now has 16 values (expanded from 11 per `front.txt`'s recommendation — added `claim_form`/`admission_note`/`diagnostic_report`/`id_proof`/`mlc`) and is the single frontend source of truth mirroring the backend's `DocumentType` enum (`backend/app/core/constants.py`); the two are kept in sync **by hand**, there's no codegen bridging Python/Pydantic and this plain TS const array, so any future enum change must be made in both places.

  **Still open**: the generated PDF report is never copied into Convex storage — `claim-results.tsx`'s "Download Report" link points straight at the backend's `GET /claims/{id}/report`, so it stops working if that claim's backend state is lost (no persistence there yet, see below). There's no retry UI if a document's `backendUploaded` flag stays `false` (the Convex-side upload always succeeds; the backend-side `fetch` failure is currently swallowed silently in `document-uploader.tsx` so one partial failure doesn't block the rest of a multi-file batch). `NEXT_PUBLIC_API_URL`/`ALLOWED_ORIGINS` (Vercel/Railway respectively) are now both set, but the end-to-end flow (create → upload → process → report) hasn't been manually verified against the live deployment yet.

- **Human Approval stage — ✅ done** (later session). Processing no longer auto-finalizes a claim: `claim.status` can now be `"awaiting_approval"` (new literal in `convex/schema.ts`'s `claims.status` union), set by `process-claim-button.tsx` when the backend's `/process` response comes back paused (`result.status === "awaiting_approval"`) instead of `completed`/`failed`. `src/components/claims/claim-approval-panel.tsx` renders exactly when `claim.status === "awaiting_approval"` — shows the AI's settlement recommendation (decision, amount, reasoning, fraud score) and three actions: Approve / Reject / Modify (with an amount input + notes). Each calls `submitDecision()` in `src/lib/backend-api.ts` (`POST /claims/{id}/decision`) then the `saveDecision` Convex mutation (`convex/claims.ts`) — which patches `officerDecision`/`officerAmount`/`officerNotes`/`officerDecidedAt` (new `claims` table fields) and flips `status` to `"completed"`, so `ClaimResults` takes over rendering. `claim-results.tsx` gained a "Final Decision (Claims Officer)" card shown whenever `claim.officerDecision` is present, alongside the existing AI-recommendation cards (unchanged — those are now understood as the AI's recommendation, not the final outcome). **Approver is the same signed-in user** (self-approval demo pattern, no role system exists yet) and the decision set is Approve/Reject/Modify-amount only (Request More Documents / Escalate to SIU deferred). Verified via the backend's network-free test suite (`backend/tests/test_claims_workflow.py`, no Sarvam credits spent).

- **Cinematic agent-pipeline overlay — added then removed** (two later sessions). A full-screen `ProcessClaimOverlay` modal was built to show the live agent timeline, but it broke in production: its `position: fixed` modal was rendered inside `process-claim-button.tsx`'s `backdrop-blur-md` container, and per the CSS spec `backdrop-filter` creates a new containing block for `fixed` descendants — the modal got trapped to that small container's box instead of the viewport, rendering as a tiny unreadable strip instead of a full-screen overlay. This also caused `claim-workspace.tsx` to hide the real `ClaimApprovalPanel`/`ClaimResults` cards while the (broken) overlay was "open," so results only appeared after navigating away and back. **Removed entirely** — `process-claim-overlay.tsx` is deleted; `process-claim-button.tsx` now shows a plain rotating "Running {agent}…" status line in the app's existing light theme while `/process` is in flight, and `claim-approval-panel.tsx`/`claim-results.tsx` render unconditionally inline (no overlay-open suppression). **Gotcha for future full-screen/modal work in this codebase**: never nest a `position: fixed` element inside an ancestor with `backdrop-filter`/`filter`/`transform`/`will-change` — those all create new containing blocks that break `fixed` positioning; portal full-screen overlays to `document.body` (or keep them outside any blurred ancestor) instead.

- **Razorpay test-mode pricing gate — ✅ manually verified end-to-end** (later session). The Free/Pro/Plus upgrade flow (`convex/payments.ts`, `/pricing` page) now works live via a real Vercel deployment click-through, not just `tsc`/`next build`. Two gotchas hit along the way: (1) `createOrder`'s Razorpay `receipt` field was built as `plan_<plan>_<full Convex user id>_<timestamp>`, which exceeded Razorpay's 40-char cap on that field and failed every order with `BAD_REQUEST_ERROR: receipt: the length must be no more than 40.` — fixed to `<plan>_<last 12 chars of user id>_<timestamp>` (well under 40 chars; the full id/plan are already recorded server-side in `pendingOrders`, so the receipt itself doesn't need to carry it). (2) Razorpay test mode rejects the generic `4111 1111 1111 1111` Visa test number with "International cards are not supported" — Razorpay India accounts have international cards disabled by default and that BIN reads as non-Indian-issued; use a Razorpay-documented domestic test card instead (e.g. Mastercard `5267 3181 8797 5449`).

- **Animated plan tag** (same session) — a small Free/Pro/Plus badge next to "Welcome, {name}" in `site-header.tsx`'s full (non-`minimal`) render, landing page (`/`) only — the stripped-down `minimal` header on `/dashboard`/`/claims/*` is untouched. New `src/components/landing/plan-tag.tsx` reuses `convex/planLimits.ts`'s `resolvePlan`/`PLAN_LIMITS` for the label (no new plan-string logic) and the existing `csShimmer`/`csPulseRing` keyframes already in `globals.css` (no new animation CSS) — Free is a quiet neutral pill, Pro shimmers emerald/cyan, Plus shimmers gold with a pulsing glow ring.

## Convex operational notes (learned the hard way this session)
- **No local Convex login for this project** — `mcp__convex__*` MCP tools always fail with "Not Authorized" here (no cached `convex.json` deployment selector). Use the CLI directly instead: `CONVEX_DEPLOY_KEY=$(grep CONVEX_DEPLOY_KEY .env | cut -d= -f2-) npx convex dev --once` pushes schema/functions to the existing prod deployment non-interactively — no `npx convex login` needed or possible in this environment.
- **Reading logs/data**: `npx convex logs` streams forever (no `--once`); wrap it in `timeout 15 npx convex logs --history 500 --jsonl` for a bounded read. `npx convex data <tableName>` (e.g. `authAccounts`, `users`) is the read-only way to inspect rows without a dashboard.
- **Deleting a test user must clear all four Convex Auth tables**, not just `users` — `authAccounts`, `authSessions`, and `authRefreshTokens` all reference `users` by ID. Deleting only the `users` row leaves an orphaned `authAccounts` row that makes every future sign-in with that same provider account throw `Uncaught Error: ... Update on nonexistent document ID` inside the `/api/auth/callback/*` HTTP action — this fails silently from the browser's perspective (redirects back to the app with no session, no error banner) and **never appears in Vercel logs** since it happens on Convex's side, not in Next.js. Only `npx convex logs` surfaces it.
- **Importing the generated Convex API from `src/`**: `convex/` lives at `frontend/` root (sibling to `src/`), not under the `@/*` tsconfig alias (`./src/*`). Import as a relative path, e.g. `../../../convex/_generated/api` from `src/components/auth/`.
- **Convex persistence — ✅ done** (later session). The backend's claim/document state no longer lives in in-memory Python dicts — `convex/schema.ts` gained `backendClaims`/`backendDocuments` tables (backend-owned, no `userId`, keyed by the backend's own `claim_id`/`document_id` strings — deliberately separate from this app's own user-scoped `claims`/`documents` tables above, same non-auth-coupled pattern as `checkpoints`/`checkpointWrites`), backed by new `convex/backendRegistry.ts` (`internalMutation`/`internalQuery`, admin-key-only) and the backend's `app/services/convex_registry.py`. Uploaded documents and generated PDF reports also moved off Railway's local disk into Convex File Storage: `convex/http.ts` gained a `POST /backend/storeFile` `httpAction` (shared-secret-guarded via a new `BACKEND_UPLOAD_SECRET` Convex env var, since `ctx.storage.store()` needs raw request bytes the backend's `convex-py` RPC client can't carry) and `convex/files.ts`'s `getFileUrl` internalQuery handles retrieval over the normal RPC path. See `backend/CLAUDE.md`'s Phase 10 registry/file-storage migration entry for the Python side. **Verified network-free only** this session (no Sarvam credits spent) — not yet exercised against a live Sarvam claim or an actual Railway restart.
- **LangGraph checkpointing — ✅ done.** `convex/schema.ts` gained `checkpoints`/`checkpointWrites` tables and `convex/checkpoints.ts` gained `internalMutation`/`internalQuery` functions (`putCheckpoint`/`putWrites`/`getLatestCheckpoint`/`listCheckpoints`/`getWrites`/`deleteThread`) — these are backend-owned, not user-facing: no `userId`, not driven by Convex Auth, keyed by the backend's own `claim_id` string as `thread_id`, and only callable by the FastAPI backend via its admin deploy key (`internalMutation`/`internalQuery` aren't exposed to the public client SDK at all). See `backend/CLAUDE.md`'s checkpointing entry for the Python side (`app/graph/checkpointer.py`, `app/services/convex_client.py`).
- **UploadThing** — file storage; the backend currently saves uploads to local disk. UploadThing wiring was deliberately deferred until this frontend existed.

## Frontend stack (as scaffolded)
Next.js 16 (App Router, `src/app/`), React 19, Tailwind v4, shadcn (`components.json`) / `@base-ui/react`, `motion`, `next-themes`, `lucide-react`. See `AGENTS.md` above — Next.js 16 has breaking changes from training-data-era Next.js; check `node_modules/next/dist/docs/` before assuming an API.

**ESLint (added this session, for CI)** — Next.js 16 removed `next lint` entirely (`next build` no longer lints either). `frontend/eslint.config.mjs` is a flat config; `eslint@^9` + `eslint-config-next@16.2.10` (pinned to match `next@16.2.10`) are devDeps, and `package.json` has `"lint": "eslint ."`. **Known bug hit and worked around**: going through `eslint-config-next`'s `FlatCompat`-wrapped `next/core-web-vitals` throws `TypeError: Converting circular structure to JSON` (`eslint-plugin-react-hooks`'s self-referencing flat config breaks `FlatCompat`'s validator) — fixed by importing `@next/eslint-plugin-next`'s flat `configs["core-web-vitals"]` directly instead of going through `FlatCompat`. Config excludes `samplecode/**` (an already-`.gitignore`d design-tool export, not app source) and `convex/_generated/**`. Also confirmed `next build` succeeds identically with or without `.env.local`/`.env` present — Convex client init (`convex-client-provider.tsx`) happens client-side at runtime, not build/prerender time, so a CI environment with zero secrets still builds. CI wiring lives in `.github/workflows/frontend.yml` at the repo root, not under `frontend/`.

Currently built: the marketing landing page (`src/app/page.tsx`, `src/components/landing/` — `hero-section`, `pipeline-showcase`, `trust-bar`, `site-header`, `aurora-background`, `animated-counter`) plus a full auth flow — sign-in/sign-up (`/sign-in`), a post-signup username step (`/username`), and Convex Auth (Password + Google) wired through the whole app. No claim submission flow or dashboard yet.
