# Things I would improve

These are not major problems—they're enhancements I'd make before calling it "production-ready."

---

# 1. Split CI into multiple workflows ⭐⭐⭐⭐⭐

Instead of

```
ci.yml
```

I recommend

```
.github/workflows/

frontend.yml

backend.yml

security.yml

release.yml
```

Why?

Suppose you only change frontend.

Current

```
Frontend

↓

Backend

↓

Tests
```

Backend still runs.

Better

```
Frontend change

↓

Only frontend workflow
```

Much faster.

---

# 2. Add Dependency Scanning

Right now you have

```
Lint

↓

Tests
```

I'd insert

```
npm audit

uv audit
```

or use GitHub's built-in Dependabot and CodeQL security scanning. GitHub provides native security features that integrate well with GitHub Actions. ([GitHub][1])

---

# 3. Cache dependencies

Instead of downloading everything every run

```
npm install
```

cache

```
node_modules

uv cache
```

GitHub Actions supports dependency caching, and `astral-sh/setup-uv` can cache packages to significantly speed up Python jobs. ([GitHub][1])

# One thing I'd change immediately

Your roadmap mentions **`next build`** for frontend validation.

That's correct.

But because you're using **Next.js 16**, remember that:

* ❌ `next lint` has been removed.
* ✅ Use the **ESLint CLI** (`eslint`) with an `eslint.config.mjs` flat configuration.
* ✅ Keep linting and build as separate CI steps. ([Next.js][2])

So your frontend job should look like:

```text
Install

↓

ESLint

↓

Type Check

↓

Build

↓

Success
```

instead of relying on build to perform linting.

# Final verdict

I would score the plan **9.8/10**.

The only thing I would change is **not** the architecture—it's already solid. I'd just make it a bit more modular and future-proof by:

* splitting CI into multiple workflows,
* adding dependency/security scanning,
* adding dependency caching,
* keeping lint and build separate for Next.js 16,
* and leaving GHCR as a later optimization.

That gives you a clean progression from a simple but production-quality CI pipeline to a more advanced release pipeline as your projects grow.

Fixed by importing `@next/eslint-plugin-next`'s flat `configs["core-web-vitals"]` directly instead of going through `FlatCompat`/`eslint-config-next`. Also excludes `samplecode/**` (an already-`.gitignore`d design-tool export, not app source) and `convex/_generated/**`.
* `frontend/package.json` — added `"lint": "eslint ."` script, `eslint` + `eslint-config-next@16.2.10` as devDependencies (pinned to match the installed Next version).
* **No backend test changes needed.** All four Sarvam-hitting tests (`test_phase9_workflow.py`, `test_medical_agent.py`, `test_policy_agent.py`, `test_document_intelligence.py`) already carry `@pytest.mark.skipif(not settings.sarvam_api_key, ...)`, and `sarvam_api_key` defaults to `""`. Since CI deliberately never sets a `SARVAM_API_KEY` secret, those tests auto-skip — no `integration` marker or `pyproject.toml` change was necessary, simpler than originally planned.

Verified locally before pushing:
* `npm run lint` → clean (0 errors after the `samplecode/` exclusion).
* `npm run build` → succeeds both with and without `.env.local` present (Convex client init happens client-side at runtime, not at build/prerender time, so missing `NEXT_PUBLIC_CONVEX_URL` in a CI environment with no secrets doesn't break the build).
* `uv run pytest -q` → **caveat**: this machine's `backend/.env` has a real `SARVAM_API_KEY`, so running it locally did NOT skip the 4 Sarvam-dependent tests — it made real Sarvam API calls (spent real credits) and 4 of them failed on a pre-existing, unrelated assertion (`test_policy_agent.py` and friends assert an agent-execution-order list that predates the `human_approval`/`report` stages added later — see `backend/CLAUDE.md`; not something this CI change introduced or something to fix here). 19 other tests passed.
